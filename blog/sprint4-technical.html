Approaching Problems

Sprint Four was a foray into problem solving. We were given a series of tests, and were told to write out a function that would allow those tests to pass. It started off simple - write a function that returns a number, or write a function that returns an array. Then it started getting a little complicated. Each function you wrote had to pass several different tests. You would write something that passed the first test, then you would adjust to meet the requirements of the second and so on.

The problem

This process went on and on, and it went well until one particular section: built in functions. JavaScript has a bunch of native classes. There's the String class, the Array class, the Number class and many many more. Each of these classes has a bunch of built in functions. For instance, the String class has a .split() function which will split the string into an array of strings based on whatever parameter you provide it. The challenge we were given was the same as the others. Use built in functions to create methods that pass the tests provided.

Simple, right?

Not for me, as it turns out.

The great thing about writing your own functions is that you can decide what information your function needs, and exactly what it should do with that information. With built in functions, you don't get to decide. It has been decided for you. Fortunately, there is documentation which describes what the function needs and what it does with that information. Unfortunately, reading those docs is a skill in itself.

One method in particular (array.map()) gave me a lot of trouble. .map() requires a function as an argument. I didn't know this, nor did I realise this while reading the docs.

Tackling the problem

I tried a variety of different problem solving methods to tackle this issue:

  • First I tried pseudocode. I wrote in plain english what I was trying to achieve. This cleared up the steps, but didn't help with exactly how to write each step
  • I tried a bunch of things. Had a guess and saw what happened. I used console.log a lot to check what information was being sent and received
  • I googled a whole lot. I found some really good starting points here, which ultimately lead to figuring out the solution

It took a long time to figure that one out, and it got a little frustration. But at least I was able to come to a solution using these techniques. Eventually.

The opposite

Sometimes (if you're lucky) when presented with a problem you have a moment of pure clarity. Everything makes sense. The mysteries of Life are open to you. Maybe that's a bit much. You know what I mean though.

I was fortunate enough to have this moment (or something like it) during the FizzBuzz challenge. The idea of the challenge is you take an array of numbers. If the number is a multiple of three, replace the number with 'Fizz'. If the number is a multiple of 5, replace it with the word 'Buzz'. If the number is a multiple of 15, replace it with the word 'FizzBuzz'. It makes total sense.

From the beginning I felt I had a pretty good idea of how to solve this problem. I started by writing some psuedocode which basically said Iterate through the array, checking each number against a condition. If the condition is met, replace the number with x and went from there. My brilliant idea was to divide the array number by 3, 5 and 15. If it returned a whole number, replace it with the appropriate FizzBuzz. Genius right? Except there's an issue. If the number is nine, well that's divisible by three so replace the number nine with fizz. But what if the number is 15? That number is divisible by 3 and 5 and 15! How will the program know which word to use?

The solution

Do it backwards.

That's really it. When you run through an if statement, the first statement that returns true causes the statement to exit, meaning the array number won't be checked against the other conditions. 15 will not be checked to see if it's divisible by 5 or 3. Only 15. I was pretty happy when that one worked.

Techniques

There are a bunch of different techniques you could use when facing a problem

  • Pseudocode
  • Trying something
  • Rubber ducky method
  • Reading error messages
  • Console.loggin
  • Googling
  • Asking people for help
  • Reflecting

Not every technique will work for each situation, but having a repertoire certainly comes in handy. After this sprint I definitely feel more confident in using these tools.
